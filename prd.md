# План Проекта

Ниже полный, практичный план для локального web-app, который собирает и показывает все `codex chats` из нескольких WSL-путей (`\\wsl.localhost\...`) с incremental-индексацией.

## Цель (MVP)
1. Локально (только на твоем ПК) собрать чаты из нескольких источников WSL.
2. Сохранить нормализованные данные в одной `SQLite` базе.
3. Дать web-интерфейс для поиска, фильтрации и просмотра диалогов.
4. Не перегружать диск: `full scan` только на первом запуске, дальше `quick/incremental`.

## Не-цели (пока)
1. Синхронизация между машинами.
2. Облачный доступ.
3. Редактирование/удаление чатов из UI.
4. Сложная авторизация (локальный режим без multi-user).

---

## Ключевые решения (зафиксировать сразу)
1. Приложение запускается на Windows-хосте.
2. Источники читаются по `\\wsl.localhost\<Distro>\home\<user>\.codex\sessions`.
3. Хранилище: `SQLite` + `FTS5`.
4. Индексация: `full scan` на первом запуске, затем `quick scan` на старте.
5. Polling во время работы делаем опциональным и выключенным по умолчанию.
6. Доступ к UI только на `127.0.0.1`.

---

## Архитектура (MVP)
1. `Source Registry`  
   Хранит список путей-источников (`Debian/labriz`, `Ubuntu/maximo` и т.д.).
2. `Scanner`  
   Обходит файлы, собирает `path / mtime / size`.
3. `Parser`  
   Читает raw-файл и извлекает сессию/сообщения в единую схему.
4. `Indexer`  
   Сравнивает с БД, определяет новые/изменённые/удалённые файлы, обновляет данные.
5. `SQLite Repository`  
   Таблицы + FTS поиск.
6. `Web API + UI`  
   Список чатов, просмотр, поиск, фильтры, ручной `Rescan`.
7. `Scheduler (optional)`  
   Polling раз в N секунд, если включен.

## Схема данных (SQLite)
1. `sources`  
   Хранит список источников (`root_path`, `distro`, `user`, `enabled`).
2. `files`  
   Отслеживание файлов (`path`, `mtime_ns`, `size`, `sha1`, `is_deleted`, `last_seen_at`, `last_indexed_at`).
3. `sessions`  
   Нормализованные сессии (`file_id`, `session_key`, `title`, `started_at`, `updated_at`, `raw_meta_json`).
4. `messages`  
   Сообщения (`session_id`, `msg_index`, `role`, `content`, `created_at`, `raw_json`).
5. `messages_fts`  
   FTS5 индекс для поиска по `content`.
6. `scan_runs`  
   История прогонов (`mode`, `started_at`, `finished_at`, `files_seen`, `files_changed`, `errors_count`, `status`).
7. `app_state`  
   Параметры приложения (`initial_scan_done`, `last_quick_scan_at`, `polling_enabled` и т.д.).
8. `parse_errors` (опционально, но полезно)  
   Ошибки парсинга с привязкой к файлу.

---

## Логика индексации (основа MVP)
1. `Full Scan` (первый запуск)  
   Проходим все файлы во всех `sources`.
2. Для каждого файла сохраняем метаданные `path + mtime + size`.
3. Парсим файл и записываем `sessions/messages`.
4. Запоминаем `initial_scan_done = true`.
5. `Quick Scan` (последующие запуски)  
   Снова обходим файлы, но сначала сравниваем только `mtime/size`.
6. Парсим только новые/изменённые файлы.
7. Для отсутствующих ранее файлов ставим `is_deleted = 1`.
8. `sha1` считаем только если `mtime/size` изменились (лениво).
9. Polling работает той же логикой, что `Quick Scan`.

---

## Стратегия парсинга (важно)
1. Сначала реализовать “robust parser” с fallback.
2. Хранить `raw_json` (или raw-фрагменты) для отладки несовместимых форматов.
3. Не падать на одном битом файле: логировать ошибку и продолжать.
4. Версионировать логику парсинга (например `parser_version` в `files` или `sessions`) на случай изменений формата.
5. Для неизвестных форматов показывать файл как “не распознан”, а не терять его.

## Безопасность и приватность
1. Слушать только `127.0.0.1`.
2. Не отправлять данные наружу.
3. Не логировать полный текст сообщений в stdout/log (только метаданные и ошибки).
4. Добавить предупреждение в UI: в базе могут храниться чувствительные данные.
5. Исключить автозагрузку приложения в сеть/публичный интерфейс.

---

## Производительность и I/O
1. На старте `quick scan`, а не `full`.
2. Polling по умолчанию `off`.
3. Хэшировать только изменившиеся файлы.
4. Использовать батчевые SQL-вставки.
5. Добавить индексы на `files.path`, `files.source_id`, `sessions.updated_at`, `messages.session_id`.
6. Ограничивать размер страницы в UI (pagination).

## План реализации по этапам (с критериями готовности)

### Этап 0. Discovery формата файлов
1. Сделать CLI-скрипт `inspect_sessions.py`.
2. Пройти по двум известным путям.
3. Вывести список файлов и пример структуры 3-5 файлов.
4. Зафиксировать реальные поля для `session/messages/timestamps`.
5. Критерий готовности: понятна структура файлов и есть примеры для тестов.

### Этап 1. База и schema
1. Создать `schema.sql`.
2. Реализовать инициализацию БД и миграцию “create if not exists”.
3. Сделать таблицы `sources/files/sessions/messages/messages_fts/app_state/scan_runs`.
4. Критерий готовности: БД поднимается локально, таблицы и индексы созданы.

### Этап 2. Scanner + Full Scan
1. Реализовать обход файлов по источникам.
2. Собирать `path, mtime_ns, size`.
3. Реализовать `full scan` с записью в `files`.
4. Критерий готовности: все файлы попадают в БД с метаданными.

### Этап 3. Parser + нормализация
1. Реализовать парсер под реальный формат `.codex/sessions`.
2. Запись `sessions` и `messages`.
3. Добавить устойчивость к ошибкам парсинга.
4. Критерий готовности: минимум 90% файлов успешно распарсены (или явно помечены ошибкой).

### Этап 4. Quick Scan / Incremental
1. Сравнение `mtime/size`.
2. Перепарсинг только изменённых/новых файлов.
3. Маркировка удалённых файлов `is_deleted = 1`.
4. Ленивый `sha1`.
5. Критерий готовности: повторный запуск не перепарсивает всё, а только изменения.

### Этап 5. Web UI (MVP)
1. Список сессий.
2. Просмотр сессии.
3. Поиск по сообщениям (FTS5).
4. Фильтры по `distro/user/date`.
5. Кнопка `Quick Rescan`.
6. Критерий готовности: можно пользоваться как локальным архивом чатов.

### Этап 6. Ops/Hardening
1. История сканов и экран ошибок.
2. Конфиг источников через файл.
3. Опциональный polling.
4. README с инструкциями запуска.
5. Критерий готовности: проект можно запустить с нуля без ручных правок кода.

## Риски и как закрыть
1. Формат файлов нестабилен  
   Решение: parser fallback + `raw_json` + `parse_errors`.
2. Медленный доступ через `\\wsl.localhost`  
   Решение: quick scan + polling off by default + ленивый hash.
3. Большая база сообщений  
   Решение: FTS5 + pagination + индексы.
4. Битые/недописанные файлы во время сканирования  
   Решение: retry 1 раз и логировать ошибку без падения всего scan.
5. Дубликаты сессий  
   Решение: `session_key` + `file_id` + дедуп политика в индексаторе.
